1. Introduction

SpendWise is a mobile application designed to help users track expenses, set budgets, monitor savings goals, and gain insight into their financial habits.

Our goal is to make personal finance easier for both individuals and groups by providing clear visual dashboards, reminders, and intelligent insights.

  • Log daily expenses quickly.
  • Create and track monthly budgets.
  • Visualize progress using Savings Circles.
  • Chat with an AI-based financial assistant.
  • Toggle between Light and Dark modes for better readability.

2. Design & Architecture

SpendWise follows an MVVM (Model–View–ViewModel) architecture to keep the UI and business logic clearly separated. The app uses:

  • ViewModel + LiveData for reactive UI updates.
  • Repository pattern to abstract data sources.
  • Firebase Firestore / Room for persistent storage.
  • SharedPreferences to cache theme and user preferences locally.
  • RecyclerView for lists like expenses and budgets.

2.1 MVVM Architecture

The Model-View-ViewModel (MVVM) architecture pattern is the foundation of SpendWise's codebase. This architectural approach provides clear separation of concerns and makes our application more maintainable, testable, and scalable.

Components of MVVM:

  • Model: Represents the data and business logic. In SpendWise, models include entities like Expense, Budget, SavingsCircle, and User. These classes encapsulate data structures and contain validation logic.
  • View: The UI layer (Activities and Fragments) that displays data to users and handles user interactions. Views observe ViewModels and update automatically when data changes.
  • ViewModel: Acts as a bridge between the View and Model. ViewModels hold UI-related data that survives configuration changes (like screen rotations) and expose LiveData objects that Views can observe. They contain no direct references to Views, making them easily testable.

Benefits of MVVM in SpendWise:

  • Separation of Concerns: Business logic is separated from UI code, making the codebase easier to understand and maintain.
  • Testability: ViewModels can be unit tested independently without Android framework dependencies, allowing for faster and more reliable tests.
  • Lifecycle Awareness: ViewModels survive configuration changes, preventing data loss when users rotate their devices.
  • Reactive Updates: LiveData provides observable data holders that automatically update the UI when data changes, reducing boilerplate code.
  • Data Consistency: Multiple Views can observe the same ViewModel, ensuring consistent data across different screens.

Throughout the SpendWise application, each major feature (Expenses, Budgets, Savings Circles, Chatbot) follows this pattern, with dedicated ViewModels managing the state and business logic for their respective features.

2.2 Code Quality with Checkstyle

To maintain high code quality and consistency across the entire codebase, SpendWise integrates Checkstyle, an automated code style checking tool. Checkstyle enforces coding standards and best practices automatically during the build process.

Checkstyle Configuration:

Our Checkstyle configuration enforces several important rules:

  • Maximum Line Length: 100 characters per line to ensure code readability and compatibility with various screen sizes and code review tools.
  • Maximum Method Length: 150 lines per method to prevent overly complex methods and encourage proper code decomposition.
  • Maximum Parameters: 7 parameters per method to maintain clean method signatures and encourage the use of parameter objects when needed.
  • Javadoc Requirements: All public classes, methods, and fields must have comprehensive Javadoc comments explaining their purpose, parameters, return values, and exceptions.
  • Naming Conventions: Enforces Java naming conventions for classes, methods, variables, and constants.
  • Code Structure: Validates proper indentation, brace placement, and whitespace usage.

Benefits:

  • Consistency: Ensures all team members follow the same coding standards, making the codebase easier to read and maintain.
  • Early Detection: Catches style violations during development, preventing code quality issues from accumulating.
  • Automated Enforcement: Removes the need for manual code review of style issues, allowing reviewers to focus on logic and architecture.
  • Documentation: Javadoc requirements ensure that code is well-documented, making it easier for new team members to understand the codebase.

Checkstyle runs automatically as part of our Gradle build process, ensuring that no code with style violations can be merged into the main branch. This automated enforcement has been crucial in maintaining code quality throughout the project's development lifecycle.

3.1 Dark Mode

SpendWise features comprehensive dark mode support, allowing users to switch between light and dark themes seamlessly throughout the entire application. This feature enhances user experience by providing better readability in low-light conditions and reducing eye strain during extended use.

Implementation Details:

  • Theme Toggle: A single toggle switch in the settings allows users to instantly switch between light and dark modes. The preference is saved using SharedPreferences, ensuring the user's choice persists across app sessions.
  • System-Wide Application: When dark mode is enabled, all screens, including Dashboard, Expense Log, Budgets, Savings Circles, Chatbot, and Profile, automatically adapt to the dark theme.
  • Consistent Color Scheme: The dark mode uses carefully selected color palettes that maintain contrast ratios for accessibility while providing a comfortable viewing experience. All UI components, including buttons, cards, text, and charts, are styled appropriately for both themes.
  • Real-Time Updates: Theme changes are applied immediately without requiring an app restart. The ViewModel architecture ensures that all active fragments and activities update their UI components reactively when the theme preference changes.
  • Persistent Storage: The user's theme preference is stored locally using SharedPreferences and synchronized with Firebase for multi-device consistency.

User Benefits:

  • Reduced eye strain in low-light environments
  • Improved battery life on devices with OLED displays
  • Personal preference accommodation for user comfort
  • Consistent visual experience across all app features

The dark mode implementation follows Material Design guidelines, ensuring that the application maintains a professional and modern appearance in both light and dark themes while providing optimal usability.

3.2 Profile & Social Features

SpendWise includes comprehensive profile management and social features that allow users to personalize their experience and connect with friends for collaborative financial management.

Profile Management:

  • Profile Access: Users can access their profile from the main navigation menu or dashboard. The profile screen displays user information, account settings, and preferences.
  • Profile Picture: Users can add, update, or change their profile picture by selecting an image from their device's gallery or taking a new photo with the camera. Profile pictures are stored securely in Firebase Storage and displayed throughout the app, including in Savings Circles and friend lists.
  • User Information: The profile screen allows users to view and edit their personal information, including name, email address, and other account details. All changes are synchronized with Firebase Authentication and Firestore database.
  • Account Settings: Users can manage various account settings from the profile screen, including notification preferences, privacy settings, and theme preferences.

Friends Feature:

  • Friend List: Users can view their list of friends from the profile screen. The friend list displays friend names, profile pictures, and their current status in shared Savings Circles.
  • Add Friends: Users can search for and add friends by email address or username. Friend requests are sent and managed through Firebase, with notifications alerting users when they receive new friend requests.
  • Friend Activity: In shared Savings Circles, users can see their friends' contributions and progress, fostering a sense of community and accountability in achieving financial goals.
  • Social Integration: The friends feature integrates seamlessly with Savings Circles, allowing users to create group savings goals with their friends and track collective progress together.

Technical Implementation:

Profile and friend data are managed through Firebase Firestore, with real-time synchronization ensuring that profile updates and friend relationships are consistent across all devices. The MVVM architecture ensures that profile and friend data are managed through dedicated ViewModels, maintaining separation of concerns and enabling efficient testing.

4.1 Unit Testing

Comprehensive unit testing is a critical component of SpendWise's development process, ensuring code reliability, preventing regressions, and maintaining high code quality throughout the application lifecycle.

Testing Strategy:

  • Business Logic Testing: All ViewModels and business logic classes are thoroughly unit tested. These tests verify that calculations, data transformations, and business rules work correctly independently of the Android framework.
  • Repository Testing: Data repository classes are tested to ensure proper data retrieval, storage, and synchronization with Firebase Firestore. Mock objects are used to simulate Firebase responses, allowing tests to run quickly and reliably.
  • Model Validation: Model classes (Expense, Budget, SavingsCircle, etc.) are tested to ensure data validation, constraint checking, and proper data structure initialization.
  • Utility Function Testing: Helper classes and utility functions, such as date formatters, currency converters, and calculation helpers, are covered by unit tests to ensure accuracy and reliability.

Testing Tools and Frameworks:

  • JUnit: The primary testing framework used for writing and executing unit tests. JUnit provides annotations and assertions that make test writing straightforward and readable.
  • Mockito: Used for creating mock objects and stubbing method calls. Mockito allows us to isolate units under test by replacing dependencies with mock implementations, ensuring tests focus on specific functionality.
  • Robolectric: Enables running Android-dependent tests on the JVM without requiring an emulator or physical device, significantly speeding up test execution times.
  • AndroidX Test: Provides additional testing utilities for Android components, including LiveData testing helpers and lifecycle-aware testing tools.

Test Coverage:

Our unit tests cover critical business logic including:

  • Expense calculation and aggregation
  • Budget tracking and threshold warnings
  • Savings Circle progress calculations
  • Data validation and error handling
  • ViewModel state management
  • Repository data operations

Benefits:

  • Early Bug Detection: Unit tests catch bugs during development, before they reach production, reducing debugging time and improving code quality.
  • Refactoring Confidence: Comprehensive test coverage provides confidence when refactoring code, ensuring that changes don't break existing functionality.
  • Documentation: Well-written unit tests serve as executable documentation, showing how classes and methods are intended to be used.
  • Regression Prevention: Tests prevent previously fixed bugs from reoccurring when new code is added or modified.

Unit tests are integrated into our continuous integration pipeline, running automatically on every commit and pull request. This ensures that all code changes are validated before being merged into the main branch, maintaining the stability and reliability of the SpendWise application.

5. Conclusions, Reflections, and Learning

Building SpendWise gave our team hands-on experience with Android development, Git-based collaboration, and clean architecture patterns.

Some of the key challenges we faced included:

  • Managing state across multiple fragments while keeping the UI responsive.
  • Ensuring theme changes (Light/Dark Mode) propagate instantly across screens.
  • Designing a database schema that works well with both budgets and expenses.

Through this project, we learned:

  • How to apply MVVM with ViewModel and LiveData in a real app.
  • How to integrate Firebase/Room and SharedPreferences effectively.
  • How to design consistent UI/UX with reusable components and themes.
